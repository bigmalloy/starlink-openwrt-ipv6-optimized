starlink-openwrt

Single-shot setup script for Starlink residential on OpenWrt 25.x

OpenWrt 25.12.0 GL-iNet Beryl AX (MT3000) Starlink residential fw4 / nftables

Quick install

scp -O starlink-setup.sh root@192.168.1.1:/tmp/
ssh root@192.168.1.1 "sh /tmp/starlink-setup.sh"

What the script does

1
IPv6 WAN

Creates or updates the wan6 interface with DHCPv6-PD, reqprefix=auto.

2
odhcpd — prefix lifetime fix

Starlink sends very short DHCPv6-PD lifetimes (~279s valid, ~129s preferred). odhcpd forwards these to LAN clients by default, causing constant address churn. The script overrides the advertised lifetimes to 3600s/7200s — the router renews internally while clients get stable addresses.

3
DNS

Disables peerdns on WAN and WAN6. Sets Cloudflare (1.1.1.1, 1.0.0.1) and Google (8.8.8.8, 8.8.4.4) resolvers with IPv6 equivalents.

4
Flow offloading

Enables software offloading, disables hardware offloading. Hardware offloading (MediaTek NPU) bypasses the Linux qdisc layer entirely, making fq_codel inactive. Software offloading keeps flows in the Linux stack so qdiscs apply.

5
MSS clamping

Sets mtu_fix 1 via uci. On OpenWrt 24.10+ (firewall4 commit 698a533), fw4 generates both ingress (mangle_forward) and egress (mangle_postrouting) clamp rules. Fixes large downloads stalling on Starlink's encapsulated link.

6
Kernel optimisation

Installs kmod-tcp-bbr (via apk on 25.x, opkg fallback). Appends a tagged sysctl block: BBR congestion control, fq_codel as default qdisc, TCP fast open, MTU probing, IPv6 forwarding, and conntrack timeout tuning. Block is replaced (not duplicated) on re-runs.

Technical notes

Prefix lifetimes vs prefix changes

The odhcpd fix addresses lifetime churn — frequent renewals where Starlink reissues the same prefix with short lifetimes. LAN clients keep their addresses while the router renews in the background. It does not prevent renumbering if Starlink assigns a genuinely new prefix (dish reboot, beam handoff). In that case, LAN clients will get new addresses regardless.

BBR scope

net.ipv4.tcp_congestion_control = bbr on the router only affects TCP sessions terminating at the router — WireGuard, OpenVPN, a local proxy or file server. It has no effect on flows from LAN clients passing through NAT. For a plain NAT router this setting is nearly a no-op.

fw4 drop-in files broken on 25.12

fw4 renders its entire ruleset as a single inline nftables script. A drop-in containing a top-level table inet fw4 block causes a syntax conflict and firewall restart fails. Do not use the drop-in approach — mtu_fix 1 is the correct fix for 24.10+.

fq_codel without a shaper

fq_codel needs back pressure to counter bufferbloat. The bottleneck on Starlink is at the satellite link, not the LAN-side ethernet port where BQL applies. Without a software shaper on the WAN side, fq_codel still provides flow hashing and some AQM benefit but cannot fully control bufferbloat. If you add SQM later, CAKE becomes the stronger choice.

Wait ~30 seconds after the script completes for IPv6 to come up.
