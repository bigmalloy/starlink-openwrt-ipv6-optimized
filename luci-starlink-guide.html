Before You BeginPrerequisites and notes

What this guide covers

Starlink residential uses DHCPv6-PD to delegate an IPv6 /56 prefix to your router. Out of the box, OpenWrt will not request this prefix or advertise it to your LAN clients. This guide walks through the LuCI screens needed to:

  • Enable IPv6 on the WAN interface via DHCPv6-PD
  • Delegate a /64 to your LAN so clients get global IPv6 addresses
  • Fix the RA interval so addresses do not expire between advertisements
  • Set reliable DNS servers (Cloudflare + Google, IPv4 and IPv6)
  • Add the Starlink dish as an NTP source
  • Enable MSS clamping to prevent oversized packets being dropped

What this guide does NOT cover

Several optimisations in starlink-setup.sh require SSH access and cannot be configured via LuCI. See Appendix A for details (accept_ra hotplug, sysctl tuning, congestion control, conntrack timeouts).

Prerequisites

  • OpenWrt 25.x running fw4 (nftables firewall)
  • Your WAN port is connected to the Starlink Ethernet adapter
  • You can reach LuCI — typically http://192.168.1.1
  • odhcpd-ipv6only package installed (included by default on OpenWrt 25.x)
⚠ Important — save after every section Click Save & Apply at the end of each section. Changes that are not applied will be lost if you navigate away or the session times out.
ℹ Preserve existing settings If you already have a working WAN connection, only modify the specific fields listed in each section. Do not change the WAN interface protocol (keep it as DHCP) unless instructed.

IPv6 WAN InterfaceNetwork › Interfaces › Add / Edit wan6

OpenWrt needs a separate wan6 interface that requests an IPv6 address and prefix from Starlink using DHCPv6. On a fresh install this interface may already exist; check before creating a new one.

Network › Interfaces

If wan6 already exists

  1. Find the row labelled wan6 and click Edit.

  2. Confirm the settings in the table below and update any that differ.

  3. Click Save & Apply.

If wan6 does not exist

  1. Click Add new interface….

  2. Set Name to wan6.

  3. Set Protocol to DHCPv6 client.

  4. Set Device to @wan (inherits the WAN physical device).

  5. Click Create interface, then fill in the fields below.

Required field values — General Settings tab

FieldValueNotes
Protocol DHCPv6 client Must be DHCPv6, not static or SLAAC
Device @wan Shares the WAN Ethernet port
Request IPv6 address try Requests a IA_NA address if available; falls back gracefully
Request IPv6 prefix auto Requests prefix delegation (Starlink provides /56)
Use DNS advertised by peer unchecked We set our own DNS in Section 4; disable auto-discovery
✓ Tip — device field @wan is an alias meaning "use the same physical device as the wan interface." This avoids hard-coding eth0 and works even if the WAN port is renamed.
ℹ Firewall zone After creating wan6, go to the Firewall Settings tab within the interface editor and assign it to the wan firewall zone. This ensures incoming IPv6 traffic is filtered correctly.

LAN Prefix AssignmentNetwork › Interfaces › LAN › Edit

Once the router receives a /56 from Starlink, it must sub-delegate a /64 to the LAN bridge so that LAN clients can obtain global IPv6 addresses. This is controlled by the ip6assign setting on the LAN interface.

Network › Interfaces › LAN › Edit

  1. Click Edit next to the LAN interface.

  2. Open the Advanced Settings tab.

  3. Find IPv6 assignment length (also labelled ip6assign in older builds).

  4. Set the value to 64.

  5. Click Save & Apply.

FieldValueNotes
IPv6 assignment length 64 Sub-assigns a /64 from the delegated /56 to br-lan
ℹ Why /64 and not /60? The official Starlink firmware uses ip6assign=60 internally, but that refers to its own internal topology. For an OpenWrt bypass router receiving a /56 via DHCPv6-PD, 64 is the correct value — it grants each LAN segment a standard /64 subnet with full SLAAC and DHCPv6 support.
✓ What to expect After saving and restarting the network (or waiting ~30 seconds), your LAN bridge should acquire a global IPv6 address such as 2404:xxxx:xxxx:0001::1/64. LAN clients will receive addresses via SLAAC or DHCPv6 from odhcpd.
3

odhcpd — RA IntervalsNetwork › DHCP and DNS › IPv6 RA Settings

Starlink delegates the /56 with short lifetimes — approximately 150 s preferred and 300 s valid. odhcpd renews the prefix every ~75 s, but LAN clients only learn about the renewed lifetime when they receive a Router Advertisement (RA). With OpenWrt's default ra_maxinterval of 600 s, the advertised preferred lifetime can expire on clients before the next RA arrives, causing them to stop using their IPv6 address for new connections.

🚨 This is the most common cause of IPv6 dropping on Starlink Symptoms: IPv6 works immediately after reboot, then fails after a few minutes on LAN clients, even though the router still has a global IPv6 address. Reducing the RA interval fixes this.
Network › DHCP and DNS › IPv6 RA Settings (or Interfaces › LAN › Edit › DHCP Server › IPv6 Settings)
ℹ Two paths in LuCI 25 In LuCI 25.x the RA interval fields may appear under Network › DHCP and DNS › IPv6 RA Settings (global), or within the LAN interface editor under the DHCP Server › IPv6 Settings tab. Use whichever path exposes RA Max Interval for the LAN interface.

  1. Navigate to the DHCP/DNS settings for the LAN interface.

  2. Locate the RA Max Interval field (UCI: ra_maxinterval).

  3. Set it to 60 seconds.

  4. Locate the RA Min Interval field (UCI: ra_mininterval).

  5. Set it to 20 seconds.

  6. Ensure RA mode is set to server and DHCPv6 mode is set to server.

  7. Click Save & Apply.

Field (UCI key)ValueNotes
RA Max Interval (ra_maxinterval) 60 Clients refresh lifetimes every ≤60 s — well within Starlink's 150 s preferred lifetime
RA Min Interval (ra_mininterval) 20 RFC 4861 §6.2.1 recommends ≥ max/3
RA mode (ra) server odhcpd sends RAs (not relay or disabled)
DHCPv6 mode (dhcpv6) server odhcpd responds to stateful DHCPv6 requests
⚠ Do not use lifetime override options Options such as max_preferred_lifetime, max_valid_lifetime, preferred_lft, and valid_lft have no effect on Starlink's prefix (it is already below the defaults) or are silently ignored. Use only the RA interval fields above.

DNS ServersNetwork › Interfaces › WAN and WAN6 › Edit

Starlink's CGNAT and CDN routing mean that using ISP-provided DNS can result in poor performance or, in some regions, incorrect responses. We disable peer DNS on both WAN interfaces and set Cloudflare and Google resolvers explicitly.

WAN interface (IPv4 DNS)

Network › Interfaces › WAN › Edit › Advanced Settings

  1. Click Edit on the WAN interface.

  2. Open the Advanced Settings tab.

  3. Uncheck Use DNS servers advertised by peer.

  4. In the Use custom DNS servers field, add each address below (one per line or comma-separated).

  5. Click Save & Apply.

NTP — Starlink Dish GPS ClockSystem › System › Time Synchronization

The Starlink dish exposes a GPS-disciplined NTP server at 192.168.100.1:123 (the management IP of the dish, reachable over the Starlink Ethernet link). It is a Stratum 1 source with ~85–123 µs accuracy — significantly better than pool.ntp.org. Available on all current Starlink hardware since mid-2024.

System › System › Time Synchronization

  1. Navigate to System › System and open the Time Synchronization tab.

  2. In the NTP server candidates list, click Add (or type into the last empty field).

  3. Enter 192.168.100.1.

  4. Leave the existing pool.ntp.org entries in place — the dish server complements them.

  5. Ensure Enable NTP client is checked.

  6. Click Save & Apply.

FieldValueNotes
Enable NTP client checked Required to use any NTP servers
NTP server (add new) 192.168.100.1 Starlink dish · Stratum 1 · GPS-disciplined
✓ Keeping existing pool servers The dish server is an addition, not a replacement. Keep the existing 0.openwrt.pool.ntp.org … 3.openwrt.pool.ntp.org entries so the router has fallback time sources if the dish is unreachable (e.g. during a firmware update).
ℹ Reachability 192.168.100.1 is only reachable over the physical WAN port that connects to the Starlink Ethernet adapter. It is not accessible from the LAN.

Software Flow OffloadingNetwork › Firewall › General Settings

Flow offloading accelerates forwarded traffic by bypassing the full Linux networking stack for established connections. OpenWrt supports two modes:

  • Software offloading — packets take a fast conntrack path inside the kernel. The qdisc layer (fq_codel) still applies.
  • Hardware offloading — packets are forwarded by the MediaTek NPU, completely bypassing the kernel qdisc. fq_codel is inactive when this is enabled.

For Starlink, software offloading is preferred: it gives near-hardware performance while keeping fq_codel active for bufferbloat control.

Network › Firewall › General Settings

  1. Open Network › Firewall.

  2. On the General Settings tab, find the Routing/NAT Offloading section.

  3. Check Software flow offloading.

  4. Leave Hardware flow offloading unchecked.

  5. Click Save & Apply.

FieldValueNotes
Software flow offloading checked (on) Fast conntrack path, qdisc still active
Hardware flow offloading unchecked (off) Would bypass fq_codel; leave disabled
ℹ Hardware offloading and fq_codel When hardware flow offloading is enabled on MediaTek-based routers (MT3000, MT7986, MT7981), the NPU handles forwarding and the Linux qdisc layer is bypassed entirely. This makes fq_codel and cake ineffective. Disable hardware offloading if you care about bufferbloat management.

MSS Clamping (mtu_fix)Network › Firewall › Zones › wan › Edit

MSS clamping prevents TCP sessions from using a segment size that, when encapsulated in IPv6, would exceed the MTU of the Starlink link. Without it, large packets may be silently dropped — causing connections to hang or stall after the initial handshake.

In OpenWrt 24.10+ (including 25.12), enabling mtu_fix on the WAN firewall zone causes fw4 to generate both ingress (mangle_forward) and egress (mangle_postrouting) clamp rules automatically. No custom nftables rules are needed.

⚠ Zone-level option only mtu_fix is a zone-level option in fw4. Setting it under General Settings › Defaults does nothing — it must be set on the wan zone specifically.
Network › Firewall › Zones

  1. Open Network › Firewall and click the Zones tab (or scroll to the zone list on the General Settings page).

  2. Find the row for the wan zone (the zone covering your WAN and WAN6 interfaces).

  3. Click Edit on the wan zone row.

  4. Locate the MSS clamping checkbox (labelled MSS clamping in LuCI, UCI key: mtu_fix).

  5. Check the box to enable it.

  6. Click Save & Apply.

Field (UCI key)ValueNotes
MSS clamping (mtu_fix) checked (enabled) Applies to the wan zone only
✓ Verifying the rules were applied After saving, SSH to the router and run:
root@OpenWrt:~# nft list chain inet fw4 mangle_postrouting | grep maxseg tcp flags syn / syn,rst tcp option maxseg size set rt mtu root@OpenWrt:~# nft list chain inet fw4 mangle_forward | grep maxseg tcp flags syn / syn,rst tcp option maxseg size set rt mtu
Both lines must appear. If only one appears, the firewall package is older than 24.10 (fw4 bug openwrt/openwrt#12112) — upgrade the firmware.
ℹ Drop-in nftables files are broken on 25.12 fw4 on OpenWrt 25.12 renders its entire ruleset as a single inline script. A top-level table block in a drop-in file under /etc/nftables.d/ causes a syntax conflict. Do not add custom clamp rules there — the mtu_fix checkbox is the correct and only method.
✅

VerificationConfirming IPv6 and MSS clamping are working

After applying all settings and clicking Save & Apply on each section, restart the network and wait ~30 seconds for DHCPv6-PD to complete.

In LuCI — Network › Interfaces

Check that each interface shows a valid address:

  • wan6 should show a global IPv6 address (e.g. 2404:xxxx::1/128 or similar)
  • LAN should show a /64 global IPv6 address (e.g. 2404:xxxx:xxxx:1::1/64)

From a LAN client

  • The client should have a global IPv6 address (not just fe80:: link-local)
  • Visit https://test-ipv6.com — should show 10/10
  • Ping: ping6 ipv6.google.com

Via SSH (deeper checks)

# WAN IPv6 address and delegated prefix root@OpenWrt:~# ip -6 addr show eth0 # Should show inet6 2404:xx... scope global # LAN delegated /64 root@OpenWrt:~# ip -6 addr show br-lan | grep "scope global" # IPv6 default route (via Starlink RA) root@OpenWrt:~# ip -6 route show default # RA intervals (should be 60 and 20) root@OpenWrt:~# uci get dhcp.lan.ra_maxinterval 60 root@OpenWrt:~# uci get dhcp.lan.ra_mininterval 20 # MSS clamp rules — both must appear root@OpenWrt:~# nft list chain inet fw4 mangle_postrouting | grep maxseg root@OpenWrt:~# nft list chain inet fw4 mangle_forward | grep maxseg # accept_ra should be 2 on eth0 (if using SSH config) root@OpenWrt:~# cat /proc/sys/net/ipv6/conf/eth0/accept_ra
⚠ accept_ra may read 0 after reboot If IPv6 stops working after a reboot, accept_ra may have been reset to 0 by netifd. This cannot be fixed via LuCI. See Appendix A for the SSH-based fix (sysctl + hotplug script).
A

Settings Not Available in LuCIRequires SSH access

The following settings are applied by starlink-setup.sh but cannot be configured through the LuCI web interface. They require SSH access to the router.

accept_ra=2 — persistent fix

Linux resets accept_ra to 0 when IPv6 forwarding is enabled (forwarding=1). This causes the router to ignore Starlink's IPv6 Router Advertisements and lose the default IPv6 route. The fix requires three layers (sysctl.conf, immediate sysctl -w, and a hotplug script) — none of which have LuCI counterparts.

# Add to /etc/sysctl.conf net.ipv6.conf.all.accept_ra = 2 net.ipv6.conf.default.accept_ra = 2 # Apply immediately root@OpenWrt:~# sysctl -w net.ipv6.conf.eth0.accept_ra=2 # Hotplug: /etc/hotplug.d/iface/25-accept_ra [ "$ACTION" = ifup ] && [ "$INTERFACE" = wan6 ] && \ sysctl -w net.ipv6.conf.eth0.accept_ra=2

Congestion control (kmod-tcp-hybla)

kmod-tcp-hybla normalises TCP congestion window growth against a 25 ms reference RTT, reducing penalty on higher-latency links. Install via:

root@OpenWrt:~# apk add kmod-tcp-hybla root@OpenWrt:~# echo "net.ipv4.tcp_congestion_control = hybla" >> /etc/sysctl.conf root@OpenWrt:~# sysctl -p

fq_codel as default qdisc

root@OpenWrt:~# echo "net.core.default_qdisc = fq_codel" >> /etc/sysctl.conf root@OpenWrt:~# sysctl -p

Conntrack timeout tuning

From the official Starlink firmware. Prevents long-lived NAT sessions (e.g. SSH tunnels, VPNs) from being dropped after the default 30-minute idle timeout.

net.netfilter.nf_conntrack_tcp_timeout_established = 7440 net.netfilter.nf_conntrack_udp_timeout = 60 net.netfilter.nf_conntrack_udp_timeout_stream = 180
